# EasyLaw 코딩 스타일

이 문서는 EasyLaw의 JavaScript/TypeScript 코드에 적용하는 기본 규칙이다.
목표는 영리한 코드보다 읽고 고치기 쉬우며, 잘못된 상태를 만들기 어려운
코드다.

## 핵심 원칙

1. **TypeScript답게 표현한다.** 타입 추론, 판별 유니온, `satisfies`,
   `async`/`await`, ES 모듈을 적극 활용한다.
2. **Go처럼 흐름을 단순하게 만든다.** 정상 흐름은 왼쪽에 두고 오류와 예외
   조건은 이른 반환으로 처리한다.
3. **Rust처럼 잘못된 상태를 타입으로 막는다.** 불변을 기본값으로 삼고,
   상태 전이는 판별 유니온과 명시적인 결과 타입으로 표현한다.
4. **Python처럼 읽기 쉬워야 한다.** 이름은 의도를 드러내고, 중첩보다 평평한
   흐름을, 축약보다 명료함을 선택한다.

## 타입과 데이터

- `const`와 `readonly`를 기본으로 사용한다. 재할당이 실제로 필요할 때만
  `let`을 사용한다.
- `any` 대신 `unknown`을 받고 경계에서 검증한다. HTTP 요청, 환경 변수,
  데이터베이스 JSON, 외부 API 응답은 신뢰하지 않는다.
- 런타임 입력은 Zod 같은 스키마로 검증한 뒤 도메인 코드에 전달한다.
- 데이터 구조와 유니온에는 `type`을 기본으로 사용한다. 선언 병합이나
  구현 가능한 공개 계약이 필요한 경우에만 `interface`를 사용한다.
- 유한한 상태는 문자열 여러 개의 조합이 아니라 판별 유니온으로 만든다.

```ts
type JobResult =
  | { status: "ready"; judgmentId: string }
  | { status: "failed"; reason: JobFailure };
```

- 선택 필드가 여러 상태를 암시하게 만들지 않는다. 타입으로 표현할 수 없는
  불변 조건은 생성 함수에서 한 번 검증한다.
- 단위가 있거나 혼동하기 쉬운 원시값은 이름으로 단위를 드러낸다.
  예: `timeoutMs`, `expiresAt`, `userId`.

## 함수와 제어 흐름

- 함수는 한 수준의 추상화와 한 가지 책임을 갖는다.
- 실패 조건과 권한 검사는 이른 반환으로 처리한다. `else` 중첩을 늘리지
  않는다.
- 같은 종류의 인수가 세 개 이상이면 이름 있는 객체 인수를 고려한다.
- 단순 변환은 `map`, `filter`, `flatMap`을 사용한다. 여러 단계의 상태 변경,
  조기 종료, 성능상 중요한 순회는 명시적인 반복문이 더 읽기 쉽다.
- 콜백 중첩보다 `async`/`await`를 사용한다. 독립 작업만 `Promise.all`로
  병렬화한다.
- 클래스는 상태와 불변 조건을 함께 캡슐화해야 할 때만 사용한다. 단순
  네임스페이스나 데이터 묶음에는 함수와 객체를 사용한다.
- 공개 함수와 API 경계는 반환 타입을 명시한다. 지역 함수는 추론이 더
  명확하면 타입을 반복하지 않는다.

## 오류 처리

- 예상 가능한 도메인 실패는 예외 대신 판별 가능한 결과로 반환한다.

```ts
type SaveResult =
  | { ok: true; id: string }
  | { ok: false; reason: "duplicate" | "unauthorized" };
```

- 예외는 I/O 실패나 불변 조건 위반처럼 정상 흐름으로 처리할 수 없는 경우에
  사용한다.
- `catch`의 값은 `unknown`으로 취급한다. 사용자에게 내부 예외 메시지나
  비밀값을 그대로 노출하지 않는다.
- 오류를 무시해야 한다면 이유를 주석으로 남기고, 가능하면 감사 로그나
  관측 가능한 신호를 만든다.

## 이름과 모듈

- 변수와 함수는 `camelCase`, 타입과 React 컴포넌트는 `PascalCase`, 상수는
  값이 진짜 전역 불변 정책일 때만 `SCREAMING_SNAKE_CASE`를 사용한다.
- 불리언은 `is`, `has`, `can`, `should`로 시작한다.
- 함수 이름은 행동을 나타낸다. `data`, `item`, `handle`처럼 문맥 없는 이름을
  공개 경계에서 사용하지 않는다.
- 파일은 기능 응집도를 기준으로 나눈다. `utils.ts`에 관련 없는 함수를
  모으지 않는다.
- 하나의 소스 파일은 원칙적으로 300줄을 넘기지 않는다. 300줄이 가까워지면
  타입, 저장소, 외부 I/O, UI 컴포넌트, 테스트 헬퍼처럼 책임 단위로 폴더를
  나눈다.
- 기능이 세 개 이상의 파일로 커지면 단일 거대 파일 대신 `feature/index.ts`
  배럴과 하위 모듈로 정리한다. 하위 모듈은 서로의 내부 구현보다 공개 함수와
  타입에 의존한다.
- 순환 의존성을 만들지 않는다. UI → 애플리케이션 → 도메인 → 인프라 방향의
  경계를 가능한 한 유지한다.

## React와 Next.js

- Server Component를 기본으로 하고, 상호작용이 필요한 가장 작은 경계에만
  `"use client"`를 둔다.
- 파생 가능한 값을 state에 복제하지 않는다. 렌더링 중 계산하거나
  `useMemo`가 실제로 필요한지 먼저 확인한다.
- Effect는 외부 시스템과 동기화할 때만 사용한다. 사용자 이벤트는 이벤트
  핸들러에서 처리한다.
- 링크와 이미지에는 Next.js가 제공하는 컴포넌트를 우선 사용한다.
- 동적 라우트와 API를 작성하기 전에 현재 설치된 Next.js 문서를 확인한다.

## 주석과 문서

- 주석은 코드가 무엇을 하는지가 아니라 왜 그렇게 해야 하는지 설명한다.
- 함수 이름으로 설명할 수 있는 주석은 함수 추출이나 이름 개선으로 바꾼다.
- 보안 경계, 데이터 우선순위, 호환성 우회에는 근거를 남긴다.
- 공개 API와 복잡한 도메인 규칙은 짧은 예제나 불변 조건을 문서화한다.

## 테스트

- 테스트 이름은 구현이 아니라 관찰 가능한 행동을 설명한다.
- Arrange → Act → Assert 흐름을 유지하고 한 테스트에서 핵심 행동 하나를
  검증한다.
- 오류 경로, 권한 경계, 중복 요청, 재시도 가능성을 정상 경로만큼 중요하게
  다룬다.
- 라우팅, 폼, 접근 제어, 반응형 레이아웃 변경에는 브라우저 테스트를
  추가하거나 실행한다.

## 변경 검증 순서

모든 코드 변경은 아래 순서를 따른다.

1. 코드를 수정한다.
2. `npm run lint`를 실행한다.
3. `npm test`를 실행한다. 필요하면 `npm run typecheck`,
   `npm run build`, `npm run test:browser`도 실행한다.
4. `npm run format`을 실행한다.
5. 기능, 코드, 파일 책임별로 변경을 나누어 Git 커밋한다.

서로 독립적인 기능과 리팩터링을 한 커밋에 몰아넣지 않는다. 포맷이 코드 의미를
바꾸는 예외가 발견되면 해당 변경을 수정한 뒤 이 검증 순서를 처음부터 다시
진행한다.
Push 작업까지 포함된 경우 push 후 GitHub Actions 실행 결과를 확인한다. 실패한
workflow가 있으면 로그를 확인하고, 현재 변경과 관련된 오류는 수정한 뒤 다시
검증한다.
